# 学生账号体系实施手册（方案 2）

## 一、目标

本手册用于指导将学生登录方式从“学号登录”切换为“独立账号登录”。

实施目标：

1. 学号只作为学生业务信息使用，不再作为登录主键。
2. 每个学生拥有唯一 `loginId`。
3. 学生首次登录必须修改初始密码。
4. 管理员可以批量生成、导出、重置学生账号。
5. 不保留旧学号登录兼容。

适用范围：

1. 网页端学生登录
2. 手机端学生登录
3. 管理员后台学生账号管理

## 二、核心规则

### 1. 登录标识

1. 学生登录使用 `loginId`。
2. `loginId` 由系统自动生成。
3. 生成规则：`S` + 8 位随机字母数字。
4. 示例：`S4K8M2Q7R`
5. `loginId` 全局唯一，不允许人工重复。

### 2. 初始密码

1. 初始密码由系统自动生成。
2. 密码长度建议为 8 到 10 位。
3. 密码必须同时包含字母和数字。
4. 不允许使用学号、姓名拼音、身份证后几位、统一默认弱密码。

### 3. 首次登录

1. 学生首次登录后必须修改密码。
2. 未修改前，不允许进入正常业务页面。
3. 修改成功后，账号进入正常使用状态。

### 4. 账号状态

1. 未生成
2. 未激活
3. 正常
4. 已禁用
5. 已锁定

### 5. 管理员权限

1. 可批量生成学生账号
2. 可批量导出学生账号
3. 可单个或批量重置初始密码
4. 可启用/禁用账号
5. 不可查看学生当前密码，只能重置

## 三、数据模型原则

### 1. 学生档案与账号分离

1. `Student` 只保存业务档案。
2. 学生账号单独保存到 `StudentAccount`。

### 2. Student 保存的内容

1. 学生姓名
2. 学号
3. 性别
4. 班级
5. 学籍状态
6. 选修班
7. 学校信息

### 3. StudentAccount 保存的内容

1. `loginId`
2. `passwordHash`
3. `enabled`
4. `locked`
5. `activated`
6. `passwordResetRequired`
7. `lastLoginAt`
8. `failedAttempts`
9. `lockedUntil`

### 4. 约束

1. 一个学生只能对应一个学生账号
2. 一个 `loginId` 只能对应一个学生账号

## 四、管理员端页面要求

### 1. 菜单结构

建议新增独立菜单：

1. 学生资料管理
2. 学生账号管理

不建议把账号管理完全混在学生资料编辑页里。

### 2. 学生账号管理列表页

列表字段：

1. 姓名
2. 学号
3. 班级
4. `loginId`
5. 账号状态
6. 是否已激活
7. 是否已改密
8. 最后登录时间
9. 最近重置时间

筛选条件：

1. 年级
2. 班级
3. 账号状态
4. 姓名/学号搜索

操作按钮：

1. 生成账号
2. 重置密码
3. 启用
4. 禁用
5. 导出账号
6. 查看详情（当前移动端已支持，网页端主要在列表内直接操作）

批量操作：

1. 批量生成账号
2. 批量重置密码
3. 批量启用
4. 批量禁用
5. 批量导出账号清单

### 3. 学生账号详情展示（当前实现）

展示内容：

1. 姓名
2. 学号
3. 班级
4. `loginId`
5. 账号状态
6. 是否首次改密
7. 最后登录时间
8. 最近重置时间

当前实现：

1. 移动端可点击列表项查看详情
2. 网页端当前暂无独立详情页，主要操作直接在列表行完成
3. 当前未开放“重新生成账号”入口，已有账号默认使用“重置密码”

禁止提供：

1. 查看当前密码
2. 导出单个学生历史密码

### 4. 账号导出页

导出字段：

1. 姓名
2. 学号
3. 班级
4. `loginId`
5. 初始密码
6. 账号状态

导出能力：

1. 按年级导出
2. 按班级导出
3. 仅导出未激活账号
4. 仅导出选中学生

导出提示：

1. 导出内容包含初始密码，请妥善保管
2. 学生首次登录后必须修改密码
3. 导出行为应记录审计日志

## 五、学生端页面要求

### 1. 登录页

字段名称：

1. 账号
2. 密码

禁止继续使用：

1. 学号
2. 学号登录

辅助说明：

1. 账号和初始密码请向学校或老师领取
2. 如果管理员刚重置过密码，请使用最新导出的初始密码登录

错误提示建议：

1. 账号或密码错误
2. 账号已禁用，请联系管理员
3. 账号已锁定，请稍后再试
4. 首次登录成功后自动跳转到改密页

### 2. 首次登录改密页

页面字段：

1. 当前账号
2. 当前密码（首次登录时填写初始密码）
3. 新密码
4. 确认新密码

密码规则提示：

1. 不少于 8 位
2. 必须包含字母和数字
3. 不能与初始密码相同

页面行为：

1. 首次登录必须先改密
2. 不允许跳过
3. 修改成功后进入系统

### 3. 忘记密码

第一阶段不开放学生自助找回密码。

统一提示：

1. 忘记密码请联系学校管理员

## 六、管理员操作步骤

### 1. 批量生成学生账号

适用场景：

1. 新学期首次启用
2. 新增一批学生后统一建账号

步骤：

1. 进入“学生账号管理”
2. 选择年级、班级或学生范围
3. 点击“批量生成账号”
4. 系统为无账号学生生成 `loginId` 和随机初始密码
5. 系统将账号状态标记为“未激活”
6. 管理员导出账号清单并发放

注意：

1. 已有账号的学生不应重复生成新 `loginId`
2. 对已有账号学生，默认应使用“重置密码”而不是“重新生成账号”

### 2. 批量导出学生账号

步骤：

1. 进入“学生账号管理”
2. 选择导出范围
3. 点击“导出账号”
4. 系统生成 Excel
5. 管理员线下或班级内发放

建议：

1. 优先按班级导出
2. 避免一次导出全校全部账号

### 3. 单个学生重置密码

步骤：

1. 搜索学生
2. 在账号列表中直接点击“重置密码”
3. 系统生成新的随机初始密码
4. 旧密码立即失效
5. 管理员需要重新导出或告知学生新初始密码
6. 学生收到新密码后重新登录并再次完成改密激活

提示文案建议：

1. 重置后旧密码将立即失效，是否继续

### 4. 启用/禁用账号

适用场景：

1. 学生离校
2. 暂停使用
3. 账号异常

规则：

1. 禁用账号后不可登录
2. 启用账号后恢复登录能力
3. 禁用账号不应影响学生业务档案

## 七、学生使用流程

### 1. 首次登录

步骤：

1. 获取账号和初始密码
2. 打开网页端或手机端登录页
3. 输入 `loginId` 和初始密码
4. 系统识别为首次登录
5. 跳转到修改密码页
6. 输入当前密码（初始密码）、新密码、确认新密码
7. 修改密码成功并自动激活账号
8. 进入系统

### 2. 日常登录

步骤：

1. 输入 `loginId`
2. 输入修改后的密码
3. 成功进入系统

### 3. 忘记密码

步骤：

1. 联系管理员
2. 管理员重置密码
3. 学生使用新初始密码重新登录
4. 学生再次完成改密后进入系统

## 八、数据迁移执行步骤

### 1. 迁移前准备

1. 确认现有学生档案完整
2. 确认学生名单、班级、学号数据可用
3. 确认管理员知道切换后不再使用学号登录
4. 确认可以向学生发放账号信息

### 2. 系统迁移动作

1. 为现有学生生成 `StudentAccount`
2. 生成 `loginId`
3. 生成随机初始密码
4. 设置
   - `enabled = true`
   - `locked = false`
   - `activated = false`
   - `passwordResetRequired = true`
5. 学生登录入口切换为新账号体系
6. 停用旧学号登录方式

### 3. 管理员迁移后动作

1. 批量导出账号清单
2. 按班级发放
3. 通知学生首次登录必须改密
4. 收集异常登录名单
5. 对个别学生执行密码重置

### 4. 学生迁移后看到的变化

1. 原学号登录失效
2. 必须使用新账号登录
3. 首次登录必须修改密码

## 九、最容易引发投诉和误操作的点

### 1. 学生不知道自己的账号

风险：

1. 学生登录失败
2. 误以为系统坏了

处理：

1. 切换前必须发放账号清单
2. 登录页明确写“账号和初始密码请联系学校领取”

### 2. 学生把学号当账号输入

风险：

1. 反复提示密码错误
2. 学生误判为密码不对

处理：

1. 登录页字段必须写“账号”
2. 不得保留“学号登录”文案
3. 发放说明中必须写明“账号不等于学号”

### 3. 管理员重复生成账号

风险：

1. 学生收到多个账号
2. 大面积混乱

处理：

1. 默认不允许对已有账号重复生成
2. 已有账号仅允许重置密码
3. 如需重新生成账号，必须二次确认

### 4. 管理员以为重置密码不会影响旧密码

风险：

1. 学生仍使用旧密码
2. 登录失败集中出现

处理：

1. 按钮写成“重置为新随机密码”
2. 操作弹窗明确提示“旧密码将立即失效”

### 5. 批量导出账号泄露

风险：

1. 账号和初始密码外泄

处理：

1. 导出操作必须记录审计日志
2. 只允许有权限管理员导出
3. 优先按班级导出，减少暴露面
4. 导出文件使用后及时清理

### 6. 切换当天学生大面积不会登录

风险：

1. 集中投诉
2. 管理员工作量骤增

处理：

1. 切换前先通知
2. 先发账号，再切换登录方式
3. 建议先选一个班级试运行

### 7. 学生首次改密失败

风险：

1. 登录后卡住
2. 误以为系统异常

处理：

1. 密码规则必须明确展示
2. 错误提示必须具体
3. 不要只提示“修改失败”

## 十、上线通知模板

### 给管理员的通知

各班学生登录方式将由“学号登录”切换为“独立账号登录”。请管理员在系统中批量生成学生账号，并按班级导出账号清单发放。切换后，学生将不再使用学号登录，首次登录必须修改密码。

### 给学生的通知

学生登录方式将调整为“账号登录”。后续请使用学校发放的账号和初始密码登录系统，不再使用学号登录。首次登录后必须修改密码。如忘记密码，请联系学校管理员重置。

## 十一、执行顺序建议

正式执行时按下面顺序推进：

1. 完成学生账号功能开发
2. 管理员端支持批量生成和导出账号
3. 管理员先生成账号
4. 小范围班级试运行
5. 修正问题
6. 全校发放账号
7. 切换学生登录入口
8. 统一处理首次登录问题
9. 稳定运行后形成常规运维流程

## 十二、运维规则

建议形成固定日常操作规范：

1. 新增学生后，必须同步生成账号
2. 学生忘记密码，一律管理员重置
3. 不允许通过口头长期复用统一默认密码
4. 学生离校或停用时，应禁用账号
5. 每次批量导出账号必须有记录
6. 首次启用期建议每天查看“未激活账号”列表

## 十三、最终推荐口径

这版手册采用的最终口径是：

1. 登录标识：系统生成 `loginId`
2. 初始密码：系统随机生成
3. 首次登录：强制改密
4. 管理员：批量导出学生账号
5. 切换策略：不兼容旧学号登录

这是一套能落地、风险可控、并且能继续演进到统一账号体系的方案。
