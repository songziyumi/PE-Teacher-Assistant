# 抢课功能剩余未完成清单（按优先级）

> 适用范围：学生抢课模块 v2  
> 当前评估：主流程已基本闭环，可试运行；以下为剩余收尾项与优化项。

## P0（上线前优先完成）

### 1. 统一教师端统计口径
- 问题：教师端部分页面仍直接使用 `currentCount` 展示“已确认人数”，可能与实际 `CONFIRMED` 唯一学生数不完全一致。
- 目标：教师端与管理端统一按“已确认唯一学生数”统计。
- 涉及文件：
  - `src/main/resources/templates/teacher/course-enrollments.html`
  - `src/main/resources/templates/teacher/courses.html`

### 2. 全量复核已确认 / 剩余 / 超编口径
- 目标：所有页面中的以下指标都统一基于同一套规则：
  - 已确认人数
  - 剩余名额
  - 超编人数
- 要求：
  - 优先使用 `CONFIRMED` 唯一学生数
  - 明确区分展示口径与缓存/计数字段

### 3. 再做一次全链路高并发回归
- 覆盖场景：
  - 第一轮抽签
  - 第二轮足球并发抢课
  - 第二轮羽毛球按班名额并发抢课
  - 退课后再次进入第二轮抢课
  - 第三轮申请与教师审批
- 验证重点：
  - 是否仍有超卖
  - 管理页与明细页统计是否一致
  - 失败提示是否准确

## P1（强烈建议尽快完成）

### 4. 教师端名单页与管理端统计字段对齐
- 目标：管理端、教师端、学生端对同一课程的展示口径一致。
- 建议：
  - 教师端列表页不要继续直接依赖 `course.currentCount`
  - 统一由服务层提供确认人数统计结果

### 5. 补一份抢课上线操作手册
- 建议包含：
  - 创建活动
  - 配置参与学生
  - 配置课程与名额
  - 开启第一轮
  - 触发抽签
  - 开启第二轮
  - 关闭活动
  - 查看报名结果与异常

### 6. 固化压测脚本与测试数据说明
- 建议整理内容：
  - JMeter 脚本用途说明
  - 第一轮 / 第二轮脚本区分
  - 学生账号 CSV 使用方法
  - 测试数据 SQL 导入步骤
- 相关目录：
  - `scripts/jmeter/`
  - `scripts/testdata/course-selection-600/`
