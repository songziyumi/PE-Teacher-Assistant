# 附件二：组织树与赛事系统角色权限矩阵

## 一、使用说明

本文档用于明确以下对象的权限边界：

1. `SUPER_ADMIN`
2. 市级 `ORG_ADMIN`
3. 区县 `ORG_ADMIN`
4. 学校 `ORG_ADMIN`
5. `TEACHER`
6. `STUDENT`

本文档解决两个问题：

1. 谁可以进入哪些菜单和接口
2. 谁可以操作哪些层级和范围的数据

原则：

1. 角色控制“功能入口”
2. 组织范围控制“数据边界”
3. 不因为是上级管理员就默认拥有下级所有明细操作权

## 二、角色定义

### 1. `SUPER_ADMIN`

定位：

1. 平台运维与系统级配置
2. 通常不参与具体市、区县、学校赛事审核链路

### 2. 市级 `ORG_ADMIN`

定位：

1. 管理一个市级组织节点
2. 可覆盖整棵市级组织子树

### 3. 区县 `ORG_ADMIN`

定位：

1. 管理一个区县组织节点
2. 可覆盖本区县所有学校

### 4. 学校 `ORG_ADMIN`

定位：

1. 管理一个学校组织节点
2. 聚焦本校教师、学生、学生账号、赛事报名

### 5. `TEACHER`

定位：

1. 本校业务执行角色
2. 负责报名草稿、队伍管理、成绩录入等执行工作

### 6. `STUDENT`

定位：

1. 个人数据查看与轻量互动角色

3. 教师端看学生数据，属于授权后的业务只读或局部可编辑，不等同于学生管理权限

## 六、典型角色场景说明

### 1. 市级管理员

允许：

1. 管理本市区县组织
2. 创建市级赛事
3. 终审报名
4. 查看全市报表

不建议默认允许：

1. 直接重置某个学校学生密码
2. 直接维护学校学生明细

### 2. 区县管理员

允许：

1. 管理本区学校组织关系
2. 创建区县赛事
3. 审核学校报名
4. 查看本区数据统计

不建议默认允许：

1. 直接增删本区学校学生账号
2. 直接代学校提交最终报名

### 3. 学校管理员

允许：

1. 管本校教师
2. 管本校学生
3. 管本校学生账号
4. 组织本校报名
5. 查看本校审核结果

### 4. 教师

允许：

1. 填报名草稿
2. 管队员信息
3. 录成绩
4. 查看本校赛事通知

不建议允许：

1. 创建组织节点
2. 审核报名
3. 管理学生账号

### 5. 学生

允许：

1. 查看本人报名信息
2. 查看本人赛事通知
3. 查看本人成绩
4. 修改本人密码

## 七、接口授权落地建议

建议把权限判定拆成三层：

1. Spring Security 负责粗角色拦截
2. 服务层负责组织范围校验
3. 核心写操作再加业务状态校验

示例：

1. “学校管理员提交报名”需要同时满足：角色正确、报名属于本校、赛事处于可提交阶段
2. “区县审核”需要同时满足：角色正确、报名属于本区、当前状态为待区县审核
3. “市级查看学生账号统计”需要同时满足：角色正确、范围属于本市、只返回统计而非敏感明细

## 八、推荐默认策略总结

建议默认策略如下：

1. 组织层级由 `ORG_ADMIN + managed_org_id` 表达，不新增一堆层级角色
2. 上级管理员默认看统计和审核，不默认做下级明细操作
3. 学生账号具体操作只留给学校管理员
4. 教师负责执行，不负责管理体系
5. `SUPER_ADMIN` 主要用于平台级兜底，不参与日常赛事业务
