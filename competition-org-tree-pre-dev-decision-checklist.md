# 组织树与赛事系统开发前决策清单

## 一、文档目的

本文档用于在正式进入组织树与赛事系统开发前，冻结必须先确认的关键决策项，避免开发过程中因口径反复变化而返工。

使用方式：

1. 每个决策项先阅读“推荐默认值”
2. 如果接受推荐方案，直接标记为“已确认”
3. 如果不接受，则从“可选方案”中选定一种
4. 对于未确认项，不建议进入对应模块的正式开发

建议规则：

1. 底座开发前，至少确认第 1、2、3、4、6、8 项
2. 成绩模块开发前，必须确认第 5 项
3. 前端并行开发前，必须确认第 7 项

## 二、决策项总览

| 序号 | 决策项 | 建议优先级 | 是否影响底座开发 |
|---|---|---|---|
| 1 | 区县和学校的真实组织数据来源 | 高 | 是 |

## 1. 区县和学校的真实组织数据来源

### 决策问题

组织树要落地，必须先明确：

1. 现有学校属于哪个区县
2. 区县名称和编码是否有统一口径
3. 学校编码是否已有稳定规则
4. 学校是否存在跨区、重复名、历史废弃数据

### 推荐默认值

推荐方案：

1. 以现有 `schools` 表为学校主数据来源
2. 单独准备一份“学校 -> 区县”映射表作为迁移输入
3. 市级节点先固定为一个顶层节点
4. 迁移脚本根据映射表批量生成 `organizations`

推荐原因：

1. 对现有系统侵入最小
2. 便于做增量迁移和回填
3. 不需要先重做学校主数据

### 可选方案

方案 A：手工整理 Excel 或 CSV 映射表后导入

1. 适合当前学校数量不大
2. 适合快速启动

方案 B：先给 `schools` 表加区县字段，再由程序生成组织树

1. 适合后续学校管理要长期维护区县字段
2. 但会增加现有学校表侵入度

方案 C：暂时只创建市级和学校级，不落区县

1. 可以更快启动
2. 但会导致后续区县审核流返工
3. 不推荐

### 需要确认的内容

1. 市级节点名称和编码
2. 区县标准名单
3. 每所学校所属区县
4. 是否存在暂不纳入组织树的学校

### 不确认的风险

1. `schools.org_id` 无法稳定回填
2. 区县管理员范围无法准确计算
3. 赛事报名审核链路无法按真实组织落地

### 结论栏

确认结果：`待确认`

## 2. 角色兼容策略最终定稿

### 决策问题

现有系统里学校管理员角色是 `ADMIN`，新方案建议统一为 `ORG_ADMIN`。需要明确兼容期怎么处理。

### 推荐默认值

推荐方案：

1. 第一阶段保留 `ADMIN`
2. 新代码同时兼容 `ADMIN` 和 `ORG_ADMIN`
3. 新增区县、市级管理员直接使用 `ORG_ADMIN`
4. 等组织树和赛事系统稳定后，再统一把 `ADMIN` 迁移成 `ORG_ADMIN`

推荐原因：

1. 对当前 Web 和移动端兼容风险最低
2. 不会一次性冲击现有管理员功能
3. 便于分阶段发布

### 可选方案

方案 A：立即统一为 `ORG_ADMIN`

1. 模型最干净
2. 但需要同步改动安全配置、前端角色判断、历史数据迁移
3. 风险较高

方案 B：长期保留 `ADMIN`，只在逻辑上视为学校级组织管理员

1. 最省事
2. 但会造成模型混乱
3. 不建议长期采用

### 需要确认的内容

1. 第一阶段是否接受双角色兼容
2. 新建学校管理员是否继续写 `ADMIN`
3. 计划何时统一迁移历史 `ADMIN`

### 不确认的风险

1. 安全配置和接口权限判断反复修改
2. 前端菜单和角色判断会不断返工
3. 测试口径不一致

### 结论栏

确认结果：`待确认`

## 3. 赛事级别边界

### 决策问题

需要明确市级、区县级、校级赛事分别由谁创建、谁发布、谁审核、谁查看。

### 推荐默认值

推荐方案：

1. 市级赛事：市级管理员创建和发布，学校报名，区县初审，市级终审
2. 区县赛事：区县管理员创建和发布，学校报名，区县审核
3. 校级赛事：学校管理员创建和发布，本校内部使用，不走区县和市级审核

推荐原因：

1. 层级清晰
2. 最符合组织树结构
3. 报名和审核链路稳定

### 可选方案

方案 A：允许市级管理员创建区县赛事模板

1. 便于统一下发标准赛事
2. 需要额外的“下发/认领”机制

方案 B：允许区县管理员创建校级赛事

1. 适合区里统一组织校内活动
2. 会让边界变复杂

### 需要确认的内容

1. 三种赛事级别是否全部要做
2. 是否允许上级代下级创建赛事
3. 是否允许草稿赛事下发到下级再补充

### 不确认的风险

1. `competition.level` 和 `host_org_id` 约束无法冻结
2. 赛事创建接口权限无法稳定
3. 后续审核链路会返工

## 4. 报名审核状态机

### 决策问题

需要明确报名从创建到最终通过的全部状态，以及驳回、撤回、重新提交的规则。

### 推荐默认值

推荐状态机：

1. `DRAFT`
2. `SUBMITTED`
3. `DISTRICT_APPROVED`
4. `DISTRICT_REJECTED`
5. `CITY_APPROVED`
6. `CITY_REJECTED`
7. `WITHDRAWN`

推荐规则：

1. 学校只能提交 `DRAFT`
2. 区县审核市级赛事报名时，可通过或驳回
3. 市级只处理区县已通过的数据
4. 报名未进入审核前允许撤回
5. 被驳回后返回草稿态再修改提交

### 可选方案

方案 A：增加 `UNDER_REVIEW` 中间态

1. 能表达“已提交但审核人尚未处理”
2. 语义更清楚
3. 但状态机会更复杂

方案 B：区县、市级都只记录审核记录，不在主表上存太多状态

1. 模型更简洁
2. 但列表筛选和前端展示会更麻烦

### 需要确认的内容

1. 驳回后是否回到 `DRAFT`
2. 是否允许学校在区县审核前撤回
3. 区县审核通过后是否还允许学校修改
4. 是否需要“补充材料中”这类状态

### 不确认的风险

1. 接口和前端按钮状态反复变化
2. 审核日志和主状态不一致
3. 历史数据补救逻辑变复杂

### 结论栏

确认结果：`待确认`

## 5. 成绩模型口径

### 决策问题

成绩字段是先统一用字符串保存，还是一开始就按计时、计距、积分等不同类型拆模型。

### 推荐默认值

推荐方案：

1. 第一阶段统一使用 `result_value` 字符串保存展示结果
2. 同时保留 `score_points`、`rank_no` 等标准字段
3. 项目类型通过 `competition_event` 配置决定校验规则
4. 等成绩模块跑通后，再考虑拆更细的结构化成绩模型

推荐原因：

1. MVP 落地快
2. 不会因成绩结构过度设计而卡住主流程
3. 便于后续按项目类型演进

### 可选方案

方案 A：一开始拆成计时、计距、得分三类字段

1. 数据更规范
2. 排名和统计更严谨
3. 但前后端复杂度显著上升

方案 B：完全只存字符串，不存可计算字段

1. 最快
2. 但后续排名、统计和导出会很麻烦
3. 不推荐

### 需要确认的内容

1. 第一阶段是否接受字符串成绩模型
2. 是否要支持自动排名
3. 是否要支持破纪录、加分规则等高级能力

### 不确认的风险

1. 成绩接口和表结构反复改动
2. 排名计算逻辑无法稳定
3. 报表导出口径不统一

## 6. 上级对学生账号的权限边界

### 决策问题

组织树落地后，市级和区县管理员对学生账号到底只能看统计，还是可以看明细、重置密码、导出账号。

### 推荐默认值

推荐方案：

1. 学校管理员拥有学生账号完整操作权限
2. 区县管理员默认只看本区统计，不看明细
3. 市级管理员默认只看全市统计，不看明细
4. 如果确有代管需求，采用显式授权开关并记录审计日志

推荐原因：

1. 保持学生账号改造时的学校边界不被破坏
2. 责任边界清楚
3. 安全和审计风险更低

### 可选方案

方案 A：区县管理员可看明细，但不可重置密码

1. 适合区级督导场景
2. 需要更细粒度的数据脱敏和导出限制

方案 B：市级、区县都可代学校操作账号

1. 管理更强势
2. 但安全风险、责任风险、误操作风险都更高
3. 不建议默认采用

### 需要确认的内容

1. 市级是否可看学校级账号明细
2. 区县是否可导出账号清单
3. 是否允许上级重置学生密码
4. 是否需要“特殊代管授权”机制

### 不确认的风险

1. 学生账号接口边界会频繁修改
2. 审计日志设计无法冻结
3. 会影响前端菜单、按钮和导出能力设计

### 结论栏

确认结果：`待确认`

## 7. Web 和移动端第一阶段范围

### 决策问题

第一阶段是只做 Web 管理端，还是同时覆盖 Flutter 管理端、教师端、学生端。

### 推荐默认值

推荐方案：

1. 第一阶段只做 Web 管理端主流程
2. 教师端如有必要，只补最小查询和录入能力
3. 移动端大规模接入放到第二阶段

推荐原因：

1. 组织树和赛事系统本身已经是大改造
2. 同时改 Web 和 Flutter 风险太高
3. 更适合先验证模型和流程是否成立

### 可选方案

方案 A：Web 管理端 + Flutter 管理端同步做

1. 用户体验更完整
2. 但联调成本高

方案 B：Web 管理端 + 教师端最小赛事能力同步做

1. 更贴近真实业务
2. 要求接口先设计得足够稳定

### 需要确认的内容

1. 第一阶段是否只面向管理端
2. 教师端是否必须首批支持报名草稿和成绩录入
3. 学生端是否第一阶段完全不接赛事能力

### 不确认的风险

1. 接口设计范围不断膨胀
2. 前端开发资源分散
3. 测试周期不可控

## 8. 上线迁移方式

### 决策问题

组织树和赛事系统要以什么方式上线，是一次性切换，还是先灰度、先隐藏入口、先只上线底座。

### 推荐默认值

推荐方案：

1. 先上线数据库增量结构和后端兼容代码
2. 先隐藏组织树和赛事入口，只给测试账号开放
3. 确认学校映射和权限范围无误后，再逐步开放管理端入口
4. 学生账号系统保持原有使用方式，不做一次性权限切换

推荐原因：

1. 风险最低
2. 便于观察迁移后的组织范围是否正确
3. 不会影响当前教学和学生账号稳定性

### 可选方案

方案 A：一次性切换上线

1. 节奏快
2. 但风险高，不建议

方案 B：只上线组织树，不开放赛事系统

1. 适合先打底
2. 最稳妥

方案 C：组织树和赛事 MVP 一起灰度上线

1. 进度更快
2. 需要测试组织足够扎实

### 需要确认的内容

1. 是否允许灰度发布
2. 是否有测试学校或测试账号
3. 是否接受第一阶段只上线底座，不开全量入口

### 不确认的风险

1. 回滚策略无法提前准备
2. 发布窗口和测试窗口无法对齐
3. 现有学校业务可能被误伤

## 三、推荐的冻结顺序

建议按以下顺序确认：

1. 先确认第 1 项：组织数据来源
2. 再确认第 2 项：角色兼容策略
3. 再确认第 3 项：赛事级别边界
4. 再确认第 4 项：报名审核状态机
5. 再确认第 6 项：学生账号权限边界
6. 再确认第 8 项：上线迁移方式
7. 然后确认第 7 项：第一阶段前端范围
8. 最后确认第 5 项：成绩模型口径

## 四、开发启动门槛

满足以下条件后，建议正式进入开发：

1. 第 1、2、3、4、6、8 项已确认
2. SQL 草案已根据确认结果更新
3. 权限矩阵已根据确认结果更新
4. 接口清单已按阶段范围收缩或扩展

若只满足部分条件，则建议仅启动底座开发，不启动完整业务页开发。
