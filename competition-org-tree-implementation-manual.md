# 组织树、赛事系统与学生账号兼容改造实施手册

## 一、文档目的

本文档用于指导现有“体育教师助手”系统，从“单学校教学管理系统”扩展为“支持市级、区县、学校三级管理的体育比赛管理系统”，并保证已经落地的学生账号体系可以继续使用，不被推翻重做。

本文档覆盖以下范围：

1. 组织树设计
2. 角色与权限模型重构
3. 赛事系统领域模型设计
4. 学生账号系统兼容改造方案
5. 数据库改造建议
6. 后端接口改造建议
7. 前端菜单与页面调整建议
8. 分阶段实施路线
9. 数据迁移策略
10. 测试与验收清单
11. 风险与回滚建议

本文档的目标不是一次性给出“最终代码”，而是给出一份可执行、可拆分、可逐步上线的实施手册。

## 二、当前系统现状

### 1. 当前角色结构

系统当前实际采用的是以角色字符串为核心的权限模型，大体包含：

1. `SUPER_ADMIN`
2. `ADMIN`
3. `TEACHER`
4. `STUDENT`

其中：

1. `SUPER_ADMIN` 用于平台级管理
2. `ADMIN` 实际承担学校管理员职责
3. `TEACHER` 用于教师端业务
4. `STUDENT` 用于学生端业务

### 2. 当前数据范围模型

当前数据边界主要建立在“学校”之上：

1. 教师和学生都绑定 `school_id`
2. 学校管理员天然只能看自己学校
3. 学生账号的权限校验也依赖学生归属学校

这意味着当前系统具备“单学校租户模型”，但不具备“跨学校、跨区县、跨市级汇总与审核”的组织层级能力。

### 3. 当前系统不足

如果直接在现有系统上继续新增角色，例如：

1. `CITY_ADMIN`
2. `DISTRICT_ADMIN`
3. `SCHOOL_ADMIN`

会出现以下问题：

1. 权限判断会在控制器和配置中不断膨胀
2. 数据过滤逻辑会散落在各个接口中
3. 比赛审批流会和角色名硬绑定
4. 后续如果新增“承办单位”“片区”“学段”会继续失控

因此本次改造的核心，不应是“继续加角色”，而应是“角色与组织范围分离”。

## 三、总体设计原则

### 1. 角色与管理范围分离

角色回答的是“你能做什么”，组织范围回答的是“你能管哪里”。

建议采用：

1. 角色控制功能权限
2. 组织树控制数据范围

### 2. 学校仍然是学生账号归属单位

学生天然属于学校，因此学生账号系统仍可继续以学校为基础单位，不需要推翻已有实现。

### 3. 赛事系统必须建立在组织树之上

赛事不是教学班级业务，天然存在：

1. 跨学校报名
2. 区县审核
3. 市级汇总
4. 多级发布

因此赛事系统必须依赖组织树进行范围控制、审批流转和统计汇总。

### 4. 分阶段改造，避免一次性重构

不建议一次性推翻现有权限体系，应采用：

1. 先补组织模型
2. 再补权限范围能力
3. 再建设赛事模块
4. 最后逐步把旧的按学校硬编码逻辑迁移到新模型

## 四、目标架构

### 1. 用户角色建议

建议将角色收敛为以下几类：

1. `SUPER_ADMIN`
2. `ORG_ADMIN`
3. `TEACHER`
4. `STUDENT`

说明：

1. `SUPER_ADMIN`：平台级运维和全局配置，不参与具体赛事层级权限
2. `ORG_ADMIN`：组织管理员，既可以是市级管理员，也可以是区县管理员或学校管理员
3. `TEACHER`：教师、教练、带队老师等执行层账号
4. `STUDENT`：学生账号

不建议长期保留 `CITY_ADMIN`、`DISTRICT_ADMIN`、`SCHOOL_ADMIN` 作为核心角色。可以在界面显示上使用这些名称，但底层统一归为 `ORG_ADMIN`。

### 2. 组织层级建议

建议建立标准组织树：

1. 市级组织
2. 区县组织
3. 学校组织

典型结构：

1. 深圳市教育局
2. 南山区教育局
3. 南山实验学校

组织层级规则：

1. `CITY` 节点无父节点
2. `DISTRICT` 节点父节点必须是 `CITY`
3. `SCHOOL` 节点父节点必须是 `DISTRICT`

### 3. 数据权限模型

每个管理员账号不再只绑定学校，而是绑定一个“管理组织节点”。

例如：

1. 市管理员的 `managedOrgId = 深圳市`
2. 区县管理员的 `managedOrgId = 南山区`
3. 学校管理员的 `managedOrgId = 南山实验学校`

权限判定原则：

1. 市管理员可看整棵市级子树下的所有学校数据
2. 区县管理员可看本区县子树下的所有学校数据
3. 学校管理员可看本校数据
4. 教师默认只看本校且与自己职责相关的数据
5. 学生只能看本人数据

## 五、组织树设计

### 1. 新增 `organizations` 表

建议新增组织表，而不是直接在 `schools` 表中叠加所有层级字段。

建议字段：

1. `id`
2. `name`
3. `code`
4. `type`，取值建议：`CITY`、`DISTRICT`、`SCHOOL`
5. `parent_id`
6. `enabled`
7. `sort_order`
8. `contact_phone`
9. `address`
10. `created_at`
11. `updated_at`

可选字段：

1. `full_name`
2. `full_path`
3. `path_codes`
4. `path_ids`

如果数据量不大，可以先不做物化路径，只保留 `parent_id`。如果后续统计很多，再补充路径字段优化查询。

### 2. 与 `schools` 的关系方案

这里有两种可行方案。

#### 方案 A：学校并入组织表

即学校不再单独作为核心组织实体，而是 `organizations` 中 `type = SCHOOL` 的节点。

优点：

1. 模型统一
2. 组织树天然完整
3. 权限过滤更简洁

缺点：

1. 现有 `School` 实体和大量业务引用改动较大

#### 方案 B：保留 `schools` 表，新增组织映射

即：

1. `organizations` 管层级
2. `schools` 继续保存学校业务属性
3. `schools.org_id` 指向 `organizations` 中的学校节点

优点：

1. 对现有代码侵入更小
2. 学生、教师、班级等实体改动较少
3. 学生账号系统几乎不用动表结构

缺点：

1. 组织树和学校表双实体并存，模型略复杂

对于当前仓库，建议优先采用方案 B。

### 3. 组织树约束规则

必须在服务层和后台管理页面同时校验：

1. 不能形成循环父子关系
2. `SCHOOL` 节点下不能再挂子节点
3. 禁用父节点时，应评估是否允许子节点继续启用
4. 删除节点前必须校验是否存在下级组织、学校、管理员、赛事数据

### 4. 组织树管理页面建议

至少包含以下功能：

1. 树形查看组织结构
2. 新增市级组织
3. 新增区县组织
4. 新增学校组织节点
5. 编辑组织名称、编码、联系人、地址
6. 启用/禁用组织
7. 关联学校实体
8. 查看当前节点下学校数、管理员数、赛事数

## 六、角色与权限模型设计

### 1. 角色不再承担层级语义

建议不要通过角色名区分市、区县、学校三个层级，而是通过组织节点类型区分。

即：

1. `ORG_ADMIN + CITY`
2. `ORG_ADMIN + DISTRICT`
3. `ORG_ADMIN + SCHOOL`

这比以下设计更稳定：

1. `CITY_ADMIN`
2. `DISTRICT_ADMIN`
3. `SCHOOL_ADMIN`

### 2. 教师与组织关系

教师仍然主要归属学校。

建议：

1. 教师保留 `school_id`
2. 如后续需要跨校执裁、带队或仲裁，可额外增加“赛事角色分配表”

不要为了少数跨校场景直接把教师主归属改成模糊组织范围，否则会影响现有教学业务。

### 3. 管理员模型建议

当前 `Teacher` 表兼容承担管理员账号。短期可继续沿用，但建议逐步补齐以下字段：

1. `role = ORG_ADMIN`
2. `managed_org_id`
3. `admin_level_snapshot`，可选，只作显示缓存

如果为了最小成本，也可以保留：

1. `ADMIN` 作为旧角色兼容值
2. 新增 `managed_org_id`

然后在业务中逐步把 `ADMIN` 迁移到 `ORG_ADMIN`。

### 4. 权限判定建议

权限判定拆成两层：

#### 第一层：入口权限

由 Spring Security 判断大类角色，例如：

1. `SUPER_ADMIN`
2. `ORG_ADMIN`
3. `TEACHER`
4. `STUDENT`

#### 第二层：数据范围权限

由业务服务层判断：

1. 当前用户管理哪个组织节点
2. 目标学校是否在该节点子树范围内
3. 目标赛事是否属于该组织范围
4. 目标报名数据是否在当前用户可见范围内

关键原则：

1. Spring Security 不负责复杂组织树数据校验
2. 数据范围校验必须收敛到公共服务类，不要散落在控制器中

### 5. 推荐的公共权限服务

建议新增类似以下服务：

1. `OrganizationScopeService`
2. `PermissionScopeService`
3. `CompetitionPermissionService`

核心能力包括：

1. 判断组织 A 是否包含组织 B
2. 判断学校是否在当前管理员范围内
3. 判断赛事是否可查看
4. 判断赛事是否可审核
5. 判断赛事是否可录入成绩

## 七、赛事系统设计

### 1. 赛事系统目标

赛事系统至少要满足：

1. 市级赛事管理
2. 区县赛事管理
3. 学校报名与队伍组织
4. 多级审核
5. 成绩录入与名次统计
6. 报表与导出

### 2. 赛事模型拆分建议

不要把“赛事”做成一个大表加几十个状态字段，建议拆分成多个领域实体。

12. `competition_start_at`
13. `competition_end_at`
14. `description`
15. `created_by`
16. `created_at`
17. `updated_at`

状态建议：

1. `DRAFT`

### 4. 赛事层级与发起主体规则

建议规则：

1. 市级赛事必须由市级组织发起
2. 区县赛事必须由区县组织发起
3. 校级赛事可由学校组织发起

派生能力：

1. 市级赛事默认面向全市区县或全市学校
2. 区县赛事默认面向本区县学校
3. 校级赛事默认面向本校内部

### 5. 审批流建议

#### 市级赛事报名流

1. 学校管理员创建报名草稿
2. 学校管理员提交报名
3. 区县管理员审核
4. 市级管理员终审
5. 审核通过后形成正式参赛名单

#### 区县赛事报名流

1. 学校管理员提交
2. 区县管理员审核通过或驳回

#### 校级赛事报名流

1. 教师或学校管理员录入
2. 学校管理员确认

### 6. 成绩录入建议

建议引入“录入”和“审核”双状态机制：

1. 裁判或录入员录入成绩
2. 赛事管理员复核成绩
3. 公布成绩后锁定结果

不要让成绩直接一录入就成为最终结果，否则后面纠错成本高。

## 八、学生账号系统兼容改造

### 1. 总体结论

学生账号系统与组织树不冲突。

原因是：

1. 学生天然归属学校
2. 学生账号只需要解决“学生如何登录”
3. 组织树解决的是“谁能管理哪些学校和赛事数据”

因此学生账号不应被纳入组织树顶层设计中重做。

### 2. 当前学生账号模型的保留部分

以下内容建议保持不变：

1. `StudentAccount` 独立于 `Student`
2. 学生账号通过 `loginId` 登录
3. 学生必须首次改密
4. 学生账号有启用、禁用、锁定、未激活等状态
5. 学生账号仍然通过 `Student -> School` 归属学校

### 3. 需要调整的地方

需要调整的是“管理学生账号的权限边界”。

当前边界默认是“当前管理员所在学校”。后续应改为：

1. 学校管理员管理本校学生账号
2. 区县管理员可查看本区县学校学生账号汇总状态，但不一定具备重置密码权限
3. 市级管理员可查看全市账号覆盖率、激活率、锁定率等汇总报表

### 4. 学生账号权限建议

建议拆成以下操作权限：

1. `studentAccount:view`
2. `studentAccount:generate`
3. `studentAccount:resetPassword`
4. `studentAccount:enable`
5. `studentAccount:disable`
6. `studentAccount:export`
7. `studentAccount:viewStats`

不同层级建议：

1. 学校管理员拥有完整账号管理权限
2. 区县管理员默认只看汇总，不直接重置学校学生密码
3. 市级管理员默认只看全局报表，不直接操作具体学生账号

如果业务上确实要求上级代管，也建议通过显式授权开关控制，而不是默认开放。

### 5. 学生账号统计建议

组织树接入后，学生账号模块可增加汇总看板：

1. 各区县学生总数
2. 已生成账号数
3. 已激活数
4. 未激活数
5. 已锁定数
6. 禁用数
7. 激活率

这些统计可以为赛事报名前的数据治理提供支持。

### 6. JWT 与登录返回兼容建议

当前登录返回中含有 `schoolId` 与 `schoolName`。建议策略如下：

1. 学生登录继续保留 `schoolId` 与 `schoolName`
2. 管理员登录新增 `managedOrgId`、`managedOrgName`、`managedOrgType`
3. 前端在过渡期继续兼容旧字段

不建议立刻删除 `schoolId`，避免移动端和网页端同时出兼容问题。

## 九、数据库改造建议

### 1. 第一阶段新增表

建议第一阶段新增以下表：

1. `organizations`
2. `competition`
3. `competition_event`
4. `competition_group`
5. `competition_registration`
6. `competition_registration_item`
7. `competition_approval_record`
8. `competition_result`
9. `competition_notice`
10. `competition_quota_rule`

### 2. 第一阶段新增字段

建议给现有表补字段：

#### `schools`

1. `org_id`
2. `district_org_id`，可选，若已通过 `org_id` 能推导则可不加
3. `city_org_id`，可选，通常不需要直接冗余

#### `teachers`

1. `managed_org_id`
2. `role` 允许新增 `ORG_ADMIN`
3. `enabled_scope_mode`，可选

#### `students`

原则上不必新增组织树字段。学生继续保留 `school_id` 即可。

### 3. 索引建议

建议增加索引：

1. `organizations(parent_id)`
2. `organizations(type, code)`
3. `schools(org_id)`
4. `teachers(managed_org_id)`
5. `competition(host_org_id, status)`
6. `competition_registration(competition_id, applicant_school_id, status)`
7. `competition_result(competition_id, competition_event_id, rank_no)`

### 4. 数据完整性约束

建议增加以下约束：

1. 一个学校必须且只能关联一个学校组织节点
2. `managed_org_id` 必须指向有效组织节点
3. 市级赛事的 `host_org_id` 必须是 `CITY`
4. 区县赛事的 `host_org_id` 必须是 `DISTRICT`
5. 报名学校必须属于赛事可覆盖范围

## 十、后端改造建议

### 1. 实体层改造顺序

建议顺序：

1. 新增 `Organization`
2. 给 `School` 增加组织关联
3. 给 `Teacher` 增加 `managedOrg`
4. 新增赛事相关实体
5. 最后才改控制器与服务层

### 2. 安全配置改造建议

当前安全配置依赖角色字符串。建议改造原则：

1. 入口层仍然保留角色控制
2. 将 `ADMIN` 逐步迁移为 `ORG_ADMIN`
3. 新增管理端接口时尽量使用 `hasAnyRole("ORG_ADMIN", "SUPER_ADMIN")`

但不要仅靠这里解决所有问题。真正的数据权限必须继续放在业务服务层。

### 3. 当前用户服务改造建议

建议增强 `CurrentUserService`，至少支持获取：

1. 当前管理员的管理组织
2. 当前管理员可见组织范围
3. 当前管理员可见学校 ID 集合

### 4. 公共范围过滤能力

建议新增公共方法：

1. `Set<Long> listSchoolIdsInManagedScope(User user)`
2. `boolean canManageSchool(User user, Long schoolId)`
3. `boolean canViewCompetition(User user, Long competitionId)`
4. `boolean canApproveRegistration(User user, Long registrationId)`

### 5. 赛事接口建议

建议按职责拆接口，而不是一个大控制器：

1. 赛事管理接口
2. 赛事项目管理接口
3. 报名管理接口
4. 审核接口
5. 成绩录入接口
6. 成绩发布接口
7. 统计报表接口

### 6. 学生账号接口改造建议

学生账号接口改造要点：

1. 查询时按组织范围汇总
2. 具体账号操作时默认只允许学校管理员
3. 如果开放区县代操作，要记录操作日志，并要求更高权限校验

## 十三、数据迁移方案

### 1. 迁移原则

迁移必须遵循：

1. 先增量，后切换
2. 先兼容，后清理
3. 先保留旧字段，确认稳定后再收缩

### 2. 组织数据初始化

建议迁移步骤：

1. 新增市级组织节点
2. 新增区县组织节点
3. 为每个学校创建一个学校组织节点
4. 将 `schools.org_id` 指向对应学校组织节点

### 3. 管理员迁移

建议迁移规则：

1. 现有 `SUPER_ADMIN` 保持不变
2. 现有学校 `ADMIN` 自动设置为对应学校组织节点的 `managed_org_id`
3. 如需新增区县管理员、市级管理员，则通过新页面创建

### 4. 学生账号迁移

学生账号本身不需要结构性迁移，只需：

1. 保留原有 `StudentAccount`
2. 补充组织范围统计查询
3. 调整管理接口的可见范围逻辑

### 5. 旧角色兼容期建议

建议兼容期内支持：

1. `ADMIN` 视为学校级 `ORG_ADMIN`
2. 新创建的管理员优先写入 `ORG_ADMIN`
3. 旧页面继续兼容 `ADMIN`

待组织树和赛事系统稳定后，再统一收口。

## 十八、实施风险

### 1. 风险一：角色和组织概念混淆

如果继续把层级直接写进角色名，后期权限会快速失控。

### 2. 风险二：控制器里到处分散写范围判断

如果不抽公共范围服务，后续接口会反复出现权限漏洞。

### 3. 风险三：一次性重构所有旧业务

这样风险太高，容易影响当前稳定的教学功能和学生账号系统。

### 4. 风险四：让上级默认直接操作下级学生账号

这会带来审计和责任边界问题。建议默认只开放汇总查看，具体账号操作仍由学校侧承担。

## 十九、推荐实施顺序总结

建议按以下顺序推进：

1. 先落组织树，不碰学生账号主体模型
2. 再做组织范围权限服务
3. 再把学生账号统计接到组织树上
4. 再做赛事系统 MVP：赛事、项目、报名、审核
5. 最后做成绩、报表、移动端、通知联动

这是对当前仓库最稳妥的路径，因为它：

1. 最大限度复用现有学校、教师、学生、学生账号体系
2. 避免一次性推翻安全模型
3. 能先把底座搭好，再逐步扩业务
