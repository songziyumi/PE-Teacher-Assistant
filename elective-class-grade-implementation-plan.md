# 选课结束后选项班年级回填实施方案

## 背景

当前在选课结束后，`班级管理` 中的选项班信息会同步生成或更新，但“年级”列直接读取 `SchoolClass.grade`，而现有同步逻辑只处理名称、类型、任课教师，没有补齐年级，因此会出现选项班年级为空的问题。

本方案的目标不是只修复列表显示，而是统一“选项班年级”的口径，避免教师端筛选、考勤、导出等后续功能再次出现数据不一致。

## 目标

- 单一年级参与的选项班，在选课结束后自动回填年级
- 跨年级参与的选项班，不错误回填某一个年级
- 统一选项班实体与学生 `electiveClass` 字段的命名规则
- 尽量兼容已有人工维护的数据

## 核心规则

### 1. 年级回填规则

- 若某课程关联的参与行政班只属于 **一个年级**，则将对应选项班 `SchoolClass.grade` 回填为该年级
- 若某课程关联的参与行政班涉及 **多个年级**，则该选项班视为“跨年级”，`SchoolClass.grade` 保持 `null`
- 若课程没有配置参与行政班，则不做强制回填，保持现状

### 2. 选项班标准命名规则

- 单一年级选项班：标准名为 `年级/班名`
- 跨年级选项班：标准名为 `班名`

### 3. 人工维护兼容规则

- 对于管理员手工创建且已经设置了年级的选项班，不做激进覆盖
- 若系统推导结果与已有年级一致，则保留
- 若系统推导结果为跨年级或与已有年级冲突，优先保守处理：不静默覆盖，记录日志或输出冲突信息

## 数据来源

唯一可信的数据来源为课程与行政班的容量关联：

- `CourseClassCapacity.course`
- `CourseClassCapacity.schoolClass`

即通过某课程关联的所有行政班，汇总这些行政班所属的 `grade`，再判断该课程对应的选项班是否属于单一年级或跨年级。

不建议从学生的 `electiveClass` 反推年级，因为该字段当前本质上是结果字段，且历史数据中并不总是包含年级前缀。

## 具体实施清单

### 一、改造 `ClassService`

文件：

- `src/main/java/com/pe/assistant/service/ClassService.java`

重点位置：

- `syncElectiveClassesFromEvent(...)`

实施内容：

- 在现有“同步选项班名称、类型、教师”的基础上，增加“根据课程参与行政班推导年级”的逻辑
- 对每个课程读取其 `CourseClassCapacity` 列表
- 汇总所有 `schoolClass.grade`
- 根据汇总结果处理：
  - 只有一个年级：给对应 `SchoolClass` 回填该年级
  - 多个年级：将该 `SchoolClass.grade` 设为 `null`
  - 无法判断：保持现状

注意事项：

- 不要删除现有的名称、类型、教师同步逻辑
- 不要从学生选课结果反推年级
- 第一版优先使用现有仓库方法 `findByCourse(course)`，保持改动最小

### 二、抽取统一的“选项班标准信息”方法

建议位置：

- 优先放在 `ClassService`
- 或抽成单独 helper/service，但第一版不建议过度重构

建议职责：

- 输入：`Course`
- 输出：
  - 是否单一年级
  - 推导出的唯一 `Grade`
  - 标准显示名/存储名

作用：

- 避免 `ClassService` 和 `StudentService` 各写一套推导逻辑
- 保证班级实体与学生 `electiveClass` 命名规则一致

### 三、改造 `StudentService`

文件：

- `src/main/java/com/pe/assistant/service/StudentService.java`

重点位置：

- `syncElectiveClassesForEvent(...)`
- `buildElectiveClassName(...)`

实施内容：

- 不再简单地把课程名直接写入 `student.electiveClass`
- 改为复用统一的“选项班标准信息”方法
- 写入规则：
  - 单一年级选项班：写入 `年级/班名`
  - 跨年级选项班：写入 `班名`

原因：

- 当前教师端和考勤端对选项班的匹配逻辑，在有年级时是按 `年级/班名` 精确匹配
- 如果只给 `SchoolClass.grade` 回填年级，而学生 `electiveClass` 仍保留旧的纯课程名，会产生新的匹配不一致问题

### 四、保留关闭活动时的同步触发点

现有触发点已经正确，不需要改流程，只需要改同步内容：

- `src/main/java/com/pe/assistant/service/CourseService.java`
- `src/main/java/com/pe/assistant/service/SelectionEventService.java`

要求：

- 保持先同步选项班，再同步学生选项班的顺序
- 避免在其他地方重复引入额外同步，减少状态分叉

### 五、调整班级管理页面展示

文件：

- `src/main/resources/templates/admin/classes.html`

实施内容：

- 年级列显示规则改为：
  - `grade != null`：显示真实年级名
  - `grade == null` 且当前班级类型为选项班：显示“跨年级”
  - 其他情况：显示 `-`

目的：

- 对管理员明确表达“该选项班不是脏数据，而是跨年级”
- 减少误判为同步失败的情况

### 六、筛选逻辑本期保持不动

文件：

- `src/main/java/com/pe/assistant/controller/AdminController.java`

说明：

- 当前班级管理的年级筛选依赖 `SchoolClass.grade`
- 第一版保持现状即可：
  - 单一年级选项班可以被年级筛选命中
  - 跨年级选项班不会出现在某个单一年级筛选结果中

原因：

- 这是符合数据语义的
- 如需增加“跨年级”筛选项，建议单独作为后续优化，不和本次修复绑定

### 七、补历史数据回补能力

建议方式：

- 增加一次性回补脚本或后台内部入口

处理范围建议：

- 类型为选项班
- 当前 `grade` 为空
- 能从已关闭活动及课程容量关系中唯一推导出年级

处理规则：

- 单一年级：自动回填
- 跨年级：不自动选某一个年级，只输出核对信息
- 已有人工年级且与推导冲突：不自动覆盖

### 八、增加日志与核对信息

建议在同步逻辑中记录：

- 本次新建了多少选项班
- 回填了多少选项班年级
- 识别出多少跨年级选项班
- 发现多少人工数据冲突

目的：

- 便于上线后核对结果
- 便于判断历史数据是否需要进一步人工处理

## 建议改动顺序

1. 抽取“选项班标准信息”统一方法
2. 改造 `ClassService.syncElectiveClassesFromEvent(...)`
3. 改造 `StudentService` 的选项班回写逻辑
4. 调整 `admin/classes.html` 展示“跨年级”
5. 增加历史数据回补能力
6. 执行编译验证

## 验证清单

### 场景一：单一年级参与

- 某课程只允许高一行政班参与
- 关闭选课后：
  - `班级管理` 中对应选项班显示 `高一`
  - 学生 `electiveClass` 为 `高一/班名`
  - 教师端学生筛选、考勤页面可正常命中

### 场景二：跨年级参与

- 某课程允许高一和高二行政班共同参与
- 关闭选课后：
  - `班级管理` 中该选项班显示“跨年级”
  - 学生 `electiveClass` 保持 `班名`
  - 不会错误归入某一个年级

### 场景三：人工维护数据

- 管理员手工创建了已带年级的选项班
- 关闭活动同步后：
  - 若系统推导一致，则保持不变
  - 若系统推导冲突，则不静默覆盖，并产生日志

### 场景四：历史数据回补

- 对已有空年级选项班执行回补后：
  - 能唯一推导年级的被正确补齐
  - 跨年级数据不被错误落库

## 编译验证

按仓库约定执行：

```bash
mvn -q -Dmaven.repo.local=.m2repo -DskipTests compile
```

## 结论

本方案的关键不是“把空年级补上”这么简单，而是统一以下三者的口径：

- 选项班实体 `SchoolClass.grade`
- 学生结果字段 `Student.electiveClass`
- 教师端 / 考勤端对选项班的匹配规则

只有这三者统一，才能真正避免后续再次出现显示正常、筛选异常、考勤不准的连锁问题。
