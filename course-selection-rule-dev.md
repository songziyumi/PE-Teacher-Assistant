# 选课抽签规则说明（开发实现版）

## 1. 文档目的

本文档用于指导后续代码改造，使第一轮抽签逻辑与目标业务规则一致。

当前阶段只定义实现目标和改造边界，不在本文档中直接落代码。

## 2. 当前实现的主要问题

当前第一轮抽签是“按课程顺序逐门处理”，而不是“按志愿优先级分阶段处理”。

这会带来以下问题：

- 课程处理顺序会影响结果。
- 第二志愿可能先于第一志愿生效。
- 第一志愿优先的业务语义不成立。
- 学生存在在同一活动第一轮内拿到多条录取记录的风险。
- “第一志愿未满但第二志愿中签”的争议场景会真实出现。

## 3. 目标实现原则

后续代码应满足以下原则：

- 第一轮按两个阶段执行：先第一志愿，后第二志愿。
- 第一轮结束后，每个学生最多 1 条 `CONFIRMED` 记录。
- 第二志愿候选集必须排除第一志愿已录取学生。
- 第二志愿容量必须基于第一志愿结算后的剩余容量。
- 第一轮结算应一次完成，不依赖课程名排序。
- 结果应可重复审计，必要时保留抽签日志或批次信息。

## 4. 推荐的目标流程

### 4.1 阶段一：第一志愿结算

- 查询当前活动中所有第一轮、第一志愿、已确认提交的记录。
- 按课程分组。
- 对每门课程执行录取：
  - 若报名人数 <= 名额，全部设为 `CONFIRMED`
  - 若报名人数 > 名额，随机抽取至名额上限设为 `CONFIRMED`
  - 未中签者设为 `LOTTERY_FAIL`
- 维护课程或班级维度的已用名额计数。

### 4.2 阶段二：第二志愿结算

- 查询当前活动中所有第一轮、第二志愿、已确认提交的记录。
- 候选集仅保留“第一志愿未录取”的学生。
- 按课程分组。
- 使用阶段一之后的剩余名额继续结算：
  - 若有效报名人数 <= 剩余名额，全部设为 `CONFIRMED`
  - 若有效报名人数 > 剩余名额，随机抽取至剩余名额上限设为 `CONFIRMED`
  - 未中签者设为 `LOTTERY_FAIL`
- 对于因第一志愿已录取而不再参与第二志愿的记录，建议设为 `CANCELLED` 或新增更清晰的状态。

### 4.3 阶段三：进入 ROUND2

- 第一轮完全结算后，活动状态切换到 `ROUND2`。
- 第二轮仍采用先到先得。
- 第一轮 `CONFIRMED` 学生可主动退课；退课后释放的名额直接进入第二轮池。
- 主动退课后，学生失去第一轮录取结果，只能按第二轮规则重新选课。
- 退课不触发第一轮补录、重算或回滚。

## 5. 数据与状态约束

基于当前实体模型，建议至少保证以下约束：

- `round = 1` 且 `status = CONFIRMED` 的记录，在同一 `event + student` 维度最多 1 条。
- `preference = 1` 和 `preference = 2` 只表示第一轮志愿位。
- `preference = 0` 继续保留给第二轮记录使用。
- 第一轮退课资格应按“是否为第一轮 `CONFIRMED` 记录”判定，不再区分第一志愿或第二志愿。

当前状态建议解释如下：

- `DRAFT`：学生已保存草稿，尚未确认提交
- `PENDING`：已确认提交，待第一轮抽签
- `CONFIRMED`：已录取
- `LOTTERY_FAIL`：该条志愿未录取
- `CANCELLED`：因规则排除、主动退课或系统调整而失效

## 6. GLOBAL 与 PER_CLASS 的实现要求

### GLOBAL 模式

- 第一志愿阶段使用课程总名额 `totalCapacity`
- 第二志愿阶段使用 `totalCapacity - 第一志愿已录取数`

### PER_CLASS 模式

- 第一志愿阶段按班级子名额独立结算
- 第二志愿阶段仍按班级剩余名额独立结算
- 学生只能占用其所属班级的名额

## 7. 推荐的实现结构

建议把第一轮抽签服务改造成以下结构：

- `loadRound1Selections(event)`
- `settleFirstPreferences(event)`
- `settleSecondPreferences(event, firstRoundConfirmedStudentIds)`
- `updateCapacityCounters(event)`
- `finalizeRound1(event)`

不要再以“遍历课程并在课程内部同时处理一志愿和二志愿”作为主流程。

## 8. 推荐的判定顺序

对每一条第二志愿记录，处理前应依次判定：

1. 学生是否已在第一志愿阶段录取
2. 该学生是否仍属于当前活动有效参与人
3. 该课程是否还有可用剩余名额
4. 该课程在对应名额维度下是否需要抽签

## 9. 建议补充的测试用例

后续改造时至少覆盖以下场景：

- 第一志愿人数未满，所有第一志愿全部录取，第二志愿仅处理剩余学生
- 第一志愿超额，第一志愿抽签后第二志愿再补录
- 学生第一志愿课程和第二志愿课程不同，第二志愿课程排序早于第一志愿课程时，仍应先看第一志愿
- 同一学生不能在第一轮得到两条 `CONFIRMED`
- GLOBAL 模式下剩余名额计算正确
- PER_CLASS 模式下班级名额不串用
- 第一志愿已录取学生的第二志愿被正确排除
- 第一轮结束后只有未录取学生能进入第二轮

## 10. 兼容与迁移注意事项

后续改造时需要评估以下兼容点：

- 管理端页面中的“志愿/轮次/状态”展示是否需要调整文案
- 历史数据中如果已存在同一学生多条 `CONFIRMED`，需要提前清洗或兼容处理
- 进度展示不应再以“当前正在处理某门课”作为唯一抽签心智模型
- 如需审计，建议记录抽签批次、随机种子或操作日志

## 11. 结论

后续代码改造目标应明确为：

- 从“按课程顺序逐门抽签”
- 调整为“按志愿优先级分阶段结算”

这能直接消除当前业务争议，并使第一志愿优先规则成立。
